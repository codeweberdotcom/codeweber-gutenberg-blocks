Обнови версию плагина codeweber-gutenberg-blocks и добавь в CHANGELOG.md записи из коммитов, которые ещё не внесены.

Рабочая директория: `wp-content/plugins/codeweber-gutenberg-blocks/`

---

# ФАЗА 1: Сбор информации

## Шаг 1: Определить текущую версию

Прочитай `plugin.php` и найди текущую версию в двух местах:
- Заголовок `* Version: X.Y.Z`
- `define('GUTENBERG_BLOCKS_VERSION', 'X.Y.Z')`

## Шаг 2: Определить новую версию

Из аргумента `$ARGUMENTS` извлеки новую версию (например `0.4.0`).
Если версия не указана — спроси пользователя, предложив варианты:
- **patch** (X.Y.Z+1) — баг-фиксы
- **minor** (X.Y+1.0) — новый функционал
- **major** (X+1.0.0) — breaking changes

Если указан флаг `--dry-run` — только показать что будет изменено, ничего не менять.

## Шаг 3: Найти последний задокументированный коммит

Прочитай `CHANGELOG.md` и определи дату/версию последней записи.
Затем выполни:

```bash
cd wp-content/plugins/codeweber-gutenberg-blocks
git log --oneline --after="<дата последней записи в CHANGELOG>" -- .
```

Если дата неточная или changelog давно не обновлялся, используй альтернативный подход:

```bash
cd wp-content/plugins/codeweber-gutenberg-blocks
git log --oneline -- . | head -50
```

И сравни коммиты с тем, что уже есть в CHANGELOG.md, чтобы найти незадокументированные.

## Шаг 4: Классифицировать коммиты

Прочитай diff каждого незадокументированного коммита (`git show --stat <hash>` и при необходимости `git show <hash>` для деталей).

Распредели изменения по категориям Keep a Changelog:
- **Added** — новый функционал, новые блоки, новые компоненты
- **Changed** — изменения существующего функционала
- **Fixed** — исправления багов
- **Removed** — удалённый функционал
- **Security** — исправления безопасности
- **Deprecated** — функционал, помеченный как устаревший

Игнорируй:
- Коммиты, которые только меняют `build/` (пересборка без изменений src)
- Merge-коммиты без содержательных изменений
- Коммиты только с изменениями документации (doc/, CLAUDE.md), если пользователь не попросит иное

## Шаг 5: Показать план

Выведи:

```
## Changelog: <текущая версия> → <новая версия>

### Новые записи:

#### Added
- ...

#### Changed
- ...

#### Fixed
- ...

### Файлы, которые будут изменены:
- plugin.php (Version header + GUTENBERG_BLOCKS_VERSION constant)
- package.json (version field)
- CHANGELOG.md (новая секция)
```

## ОБЯЗАТЕЛЬНАЯ ПАУЗА

**НЕ МЕНЯЙ ФАЙЛЫ.** Спроси пользователя:

> Применить эти изменения? Можешь отредактировать записи или изменить версию.

Жди ответа. Если пользователь хочет скорректировать — внеси правки в план и снова покажи.
Переходи к Фазе 2 ТОЛЬКО после явного подтверждения.

# ФАЗА 2: Применение (только после подтверждения)

## Шаг 6: Обновить версию в plugin.php

Замени версию в двух местах:
1. `* Version:           <новая версия>` (в заголовке плагина)
2. `define('GUTENBERG_BLOCKS_VERSION', '<новая версия>');`

## Шаг 7: Обновить версию в package.json

Замени `"version": "<старая>"` на `"version": "<новая>"`.

## Шаг 8: Обновить CHANGELOG.md

Добавь новую секцию **сразу после строки `---` (после описания файла, перед первой записью)**.

Формат новой секции:

```markdown
## [<новая версия>] - <сегодняшняя дата YYYY-MM-DD>

### Added
- **Блок/Компонент**: описание

### Changed
- **Блок/Компонент**: описание

### Fixed
- **Блок/Компонент**: описание
```

Не используй эмодзи в заголовках секций (в отличие от старых записей — приведи к единому стилю без эмодзи).

## Шаг 9: Итоговый отчёт

Выведи:

```
## Готово: версия обновлена <старая> → <новая>

### Изменённые файлы:
- plugin.php — версия в заголовке и константе
- package.json — version
- CHANGELOG.md — добавлена секция [<новая>]

### Следующие шаги:
- Проверь CHANGELOG.md — убедись что записи корректны
- Запусти `npm run build` для пересборки
- Закоммить: `release: bump version to <новая>`
```

## Шаг 10: Предложить коммит

Спроси:

> Закоммитить обновление версии? (`release: bump version to <новая>`)

Если пользователь подтвердит — выполни `/commit`.
