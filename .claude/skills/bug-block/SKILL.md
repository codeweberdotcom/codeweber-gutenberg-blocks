Диагностируй и исправь баг в Gutenberg-блоке плагина codeweber-gutenberg-blocks.

Рабочая директория плагина: `wp-content/plugins/codeweber-gutenberg-blocks/`

---

# ФАЗА 1: Анализ

## Шаг 1: Определить блок

По аргументу `$ARGUMENTS` определи название блока и описание проблемы.
Найди папку блока в `src/blocks/<block-name>/`.
Если из описания непонятно какой блок — спроси пользователя.

## Шаг 2: Прочитать файлы блока

Прочитай все файлы блока в `src/blocks/<block-name>/`:
- `block.json` — атрибуты (источник правды)
- `edit.js` — логика редактора
- `save.js` — статический вывод на фронте
- `render.php` — динамический вывод (если есть)
- `style.scss` / `editor.scss` — стили
- `index.js` — регистрация, deprecated

Если блок использует shared-компоненты из `src/components/` или утилиты из `src/utilities/` — прочитай и их.

## Шаг 3: Диагностика

Проанализируй код в контексте описанной проблемы. Типичные причины:
- Неправильные Bootstrap-классы или `data-bs-*` атрибуты
- Ошибки в условном рендеринге (edit.js или save.js)
- Атрибуты, которые не прокидываются из edit в save/render
- Неправильные селекторы или свойства в SCSS
- Ошибки в PHP-рендере (динамические блоки)
- Баги в shared-компонентах, затрагивающие этот блок

## Шаг 4: Отчёт

Выведи структурированный отчёт:

```
## Диагностика: <название блока>

### Проблема
Краткое описание бага своими словами.

### Причина
Что именно вызывает баг, в каком файле и на какой строке.

### План исправления
1. Конкретное изменение №1 (файл, что поменять)
2. Конкретное изменение №2 ...
3. Нужен ли deprecated (да/нет и почему)

### Риски
Может ли фикс затронуть другие блоки или поведение.
```

## ОБЯЗАТЕЛЬНАЯ ПАУЗА

**НЕ ТРОГАЙ КОД.** Спроси пользователя:

> Приступить к исправлению по этому плану?

Жди ответа. Если пользователь хочет изменить план — скорректируй и снова спроси подтверждение.
Переходи к Фазе 2 ТОЛЬКО после явного подтверждения.

---

# ФАЗА 2: Исправление (только после подтверждения)

## Шаг 5: Исправить код

Выполни изменения строго по утверждённому плану. Минимальные правки, без рефакторинга.

Если исправление меняет HTML-вывод `save.js`:
- Скопируй старую версию `save` в deprecated-запись в `index.js`
- Добавь `isEligible` и `migrate` если менялись атрибуты
- Образец deprecated-записи — смотри `src/blocks/tables/index.js`

## Шаг 6: Lint

```bash
cd wp-content/plugins/codeweber-gutenberg-blocks
npm run lint:js
```

Если ошибки уровня `error` — исправь и повтори.

## Шаг 7: Build

```bash
npm run build
```

Если ошибки — исправь и повтори.

## Шаг 8: Отчёт об исправлении

Выведи:

```
## Исправлено: <название блока>

### Что изменено
- файл:строка — описание изменения

### Deprecated
Добавлен / не нужен (причина).

### Что проверить
- Шаги для проверки фикса в браузере
- Нужно ли пересохранять посты в редакторе
```

## Шаг 9: Предложить коммит

Спроси пользователя:

> Закоммитить исправление? (fix: <краткое описание>)

Если пользователь подтвердит — выполни `/commit`.
